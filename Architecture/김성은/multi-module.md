## 멀티 모듈이란?
배포 단위는 하나로 유지하되, 의미 있는 경계를 기준으로 코드를 분리하는 구조화 전략

## 프로젝트 구조 비교
- **모놀리식**: 하나의 프로젝트, 하나의 빌드, 하나의 배포
- **멀티 모듈**: 배포는 하나, 코드는 분리
- **MSA**: 서비스마다 DB, 서버, 배포 다름

## 멀티모듈의 장점
- 도메인 중심 설계 가능 -> 비즈니스 규칙이 서비스/컨트롤러에 흩어지지 않음
- 의존성 방향이 컴파일 단계에서 강제됨
- 테스트 속도, 신뢰도, 유지보수성이 향상
- 팀 협업이 편해짐 (충돌 감소)

## 테스트 전략
- domain → 순수 단위 테스트
- application → 유스케이스 테스트
- infrastructure → 통합 테스트

## 멀티 모듈 구조를 나누는 기준
- 하나의 책임과 하나의 규칙을 스스로 설명할 수 있는 경계를 나누는 것이 가장 중요
- 기준 질문
	1. 이 모듈은 무엇을 책임지는가?
	2. 이 모듈은 혼자 설명 가능한가?
	3. 이 모듈은 누구에게 의존하는가?

## 예시 멀티모듈 구조

```
root
 ├─ app-api        # 실행 모듈 (Controller, Entry Point)
 ├─ application    # 유스케이스, 서비스 로직
 ├─ domain         # 핵심 비즈니스 규칙
 ├─ infrastructure # DB, 외부 API 연동
 └─ common         # 공통 유틸, 예외
```

## 핵심 정리
- 멀티모듈의 핵심은 기술 분리가 아니라 책임과 규칙이 스스로 설명 가능한 경계를 정의하는 것
- domain 모듈이 기술(Spring, JPA)에 의존하지 않도록 주의
